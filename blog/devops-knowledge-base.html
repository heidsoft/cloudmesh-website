我是如何用 AI 整理 8 年 DevOps 笔记的

从散乱文件夹到有序知识库，一天完成，AI 帮了大忙

2026年5月9日 8 分钟阅读

昨天花了一整天，把 GitHub 上一个积攒了 8 年的 heidsoft/devops 仓库整理了一遍。30+ 个散乱的文件夹，中文英文混在一起，PDF 和 Markdown 混在一起，有的内容已经过时，有的到现在还有价值。

这篇文章说说我是怎么整理的，思路和工具是什么，以及 AI 在里面帮了哪些忙。

一、问题：为什么知识库会变成"知识垃圾堆"？

DevOps 工程师的笔记本里，通常会堆积这些东西：

  • 学 K8S 时的安装命令截图
  • 2015 年搭 Docker 集群的运维日志
  • 某个客户现场的踩坑记录
  • 读了一半的 PDF 电子书
  • 同事发来的学习资料，转发两次就找不到了

时间一长，文件夹越来越多，命名越来越随意，找东西全靠"我记得有这么个文件"。

知识的积累 ≠ 知识的组织。
积累是本能，组织是设计。

二、整理思路：三层分类法

这次整理，我用了三层分类：

第一层：按用途分

📚 _docs/

技术文档（按主题分类）

devops / architecture / cloud / middleware / network / frontend / linux / db / git

💻 _code/

代码（源码 + 重构示例）

src/ / code-refactoring/

🎨 _design-patterns/

设计模式学习资料

UML / 状态机 / 职责链

📦 _archive/

归档（废弃内容 / 大 PDF / 图片）

明确标注"不再维护"

第二层：按主题分

每个大类下面，再按细分主题分。比如 _docs/devops/ 下面有：

  • docker/ — Docker PDF 教程 20+ 篇
  • k8s/ — Kubernetes PDF 资料
  • AI-DevOps/ — AI 运维智能化的框架思考
  • paas/ — PaaS 白皮书

第三层：每个目录必须有 README

这一步最关键。目录里有没有 README，决定了这个目录的内容会不会被再次利用。

⚠️ 血的教训

之前每次整理完，过三个月再看，又乱回来了。

原因：没有在每个目录放 README，新文件不知道往哪放。

这次立了规矩：新增文件必须同步更新 README 索引。

三、AI 帮了哪些忙？

说起来，这次整理全程是 AI 辅助的。具体做了这些：

1. 快速扫描 + 理解内容

30+ 个文件夹、300+ 个文件，靠人工一个个看太慢。AI 帮忙把每个文件夹的 .md 文件扫了一遍，提取出核心主题和精华摘要。

2. 生成目录 README

每个子目录的 README 不是简单列文件清单，而是包含：

  • 这个目录的主题是什么
  • 每个文件的说明（不是文件名，是内容简介）
  • 精华片段摘录（命令示例、核心概念）
  • 和其他目录的关联

3. 识别过时内容

扫了一遍之后，K8S v0.17 安装（2015年的）、Boot2Docker 命令这些技术已经迭代的内容，打上"仅供参考"的标签，归到 _archive/。

四、整理后的目录结构

devops/
├── _docs/
│   ├── devops/         ← 容器化·K8s·Spring·鉴权
│   ├── architecture/   ← SOA/ESB/高可用
│   ├── cloud/          ← OpenStack/CloudStack/ZStack
│   ├── middleware/    ← RabbitMQ/Nginx/HAProxy
│   ├── network/       ← SDN/iptables
│   ├── frontend/      ← AngularJS/CSS/DataTables
│   ├── linux/         ← libvirt/vagrant/内核
│   ├── db/            ← 哈希表/存储
│   ├── git/           ← Git 版本控制
│   └── products/       ← 产品设计文档（新！）
├── _code/             ← Go 源码 + 重构示例
├── _design-patterns/   ← 设计模式
└── _archive/          ← 归档（大PDF/过时内容）

五、README 规范：让知识可被发现

这是最值得分享的一条经验：

README 不是装饰，是知识的入口。
好的 README = 一句话说明 + 文件索引 + 精华摘要 + 可点击链接

具体到每个子目录的 README，我用了这个模板：

# 目录主题

> 一句话说清楚这个目录是做什么的

## 📄 笔记文件

| 文件 | 说明 |
|------|------|
| xxx.md | 这篇讲了什么 |

## 📁 子目录

- `docker/` — Docker PDF 教程（20+ 篇）

## 🔥 精华摘要

核心命令/概念摘录...

六、后续维护机制

整理一次容易，保持整洁难。这次我设了几个规矩：

维护规则

  • 新增文件 → 必须同步更新对应 README 的索引
  • 过时内容 → 移到 _archive/，不要直接删
  • 每月底 → 检查是否有孤儿文件（有了文件但没加索引）

七、工具链

工具用途
GitHub知识库托管，版本历史可追溯
AI 辅助阅读快速理解每个文件的内容
Markdown所有笔记用 Markdown，搜索友好
正则批量替换修复 45 个博客文章的 CSS 引用问题

结语

整理知识库这件事，最难的不是技术，而是决心。

每次想整理的时候，总觉得"等有空再说吧"。结果越积越多，最后变成历史包袱。

这次用 AI 辅助，一天完成了本来可能拖一周的工作。AI 不能替你做判断（哪些保留，哪些归档），但能做执行（读文件、写摘要、改结构）。

让自己的知识真正可复用、可搜索、可传承，这可能是对自己时间最好的一笔投资。
