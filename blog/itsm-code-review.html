🔍 ITSM 开源项目代码质量审查实战

Go + Next.js 项目踩坑记录

2026-03-17 · 阅读时间：8 分钟

Go Next.js 代码审查 开源

📖 背景

最近在维护 ITSM 开源项目，这是一个企业级 IT 服务管理系统，采用 Go + Next.js 技术栈。为了确保代码质量，我进行了一次全面的代码审查。

项目规模：

  • 后端：Go 1.25 + Gin + Ent ORM
  • 前端：Next.js 15 + React 19 + Ant Design
  • 代码量：2000+ 文件变更

🚨 发现的问题

1. Go 版本兼容性问题

🔴 高优先级

go.mod 要求 Go >= 1.25.6，但服务器运行的是 Go 1.25.5

go: go.mod requires go >= 1.25.6 (running go 1.25.5; GOTOOLCHAIN=local)

解决方案：升级 Go 版本或修改 go.mod 中的版本要求。

2. panic 不当使用

🔴 高优先级

在 auth_service.go 中使用 panic 处理错误

// 错误示例 if err != nil { panic("failed to generate random token: " + err.Error()) } // 正确做法 if err != nil { return nil, fmt.Errorf("failed to generate random token: %w", err) }

3. 未完成的 TODO

🟡 中优先级

代码中存在多个未实现的功能

文件 行号 待完成
incident_escalation_service.go 251 实现通知发送逻辑
sla_policy_service.go 221 实现业务时间计算
root_cause_analysis_service.go 51, 181 集成 AI 分析服务

4. 前端类型问题

🔴 高优先级

缺少 @ant-design/icons 类型声明（90+ 文件受影响）

error TS2307: Cannot find module '@ant-design/icons' or its corresponding type declarations.

解决方案：运行 pnpm add @ant-design/icons 安装依赖。

✅ 做得好的地方

值得肯定的设计

  • 🔐 认证安全：JWT + Refresh Token 双令牌机制，密码使用 bcrypt 哈希
  • 🏢 多租户架构：完善的租户隔离设计，数据安全有保障
  • 🛡️ 错误信息：敏感信息不泄露，如登录失败统一返回"用户名或密码错误"
  • 📦 资源管理：大部分数据库连接、文件句柄都有 defer Close()
  • 🔄 Panic 恢复：BPMN 服务有 recover 机制，防止意外崩溃

📋 代码审查清单

分享一个适用于 Go + React 项目的审查清单：

  1. 安全性
    • SQL 注入风险（优先使用 ORM）
    • 敏感信息硬编码
    • 认证授权完整性
  2. 错误处理
    • 避免 panic，使用 error 返回
    • 错误信息不要泄露敏感数据
    • 使用 fmt.Errorf 包装错误
  3. 资源管理
    • defer Close() / defer Unlock()
    • context 传递而非 context.Background()
    • goroutine 泄漏检查
  4. 代码质量
    • TODO/FIXME 清理
    • console.log 移除
    • any 类型替换为具体类型

🔧 实用命令

# 后端检查 go vet ./... go test ./... -race -cover # 前端检查 pnpm tsc --noEmit pnpm eslint src --max-warnings=0 # 查找问题代码 grep -rn "TODO\|FIXME\|panic\|console.log" ./src

📚 总结

代码审查是保证项目质量的重要环节。对于开源项目来说，良好的代码质量不仅能减少 Bug，还能吸引更多贡献者参与。

建议：

  • 定期进行代码审查，不要积累太多技术债务
  • 使用自动化工具（CI/CD）检查代码质量
  • 及时清理 TODO 和调试代码
  • 编写单元测试，提高覆盖率
