🔥 深度 AI Agent 架构设计 Hermes

Nous Research Hermes Agent 架构深度解析：自我进化 AI Agent 的工程实践

2026年4月13日 约 20 分钟 CloudMesh Team
深入解析 Hermes Agent 的核心架构设计：自我进化机制、Provider 中立架构、Tool Registry、Skills 系统、多平台网关，以及对国产 AI Agent 发展的启示。

🤖 引言：为什么研究 Hermes Agent

过去一年，AI Agent 概念火热，但大多数项目停留在「 demo 级别」——能跑，但无法真正用于生产。Hermes Agent 来自 Nous Research，是目前少数真正投入生产的开源 AI Agent 项目之一。

它最核心的差异化是什么？自我进化能力——Agent 不是每次都是新手，而是能从历史经验中持续学习和优化。

Hermes Agent 核心定位

"The self-improving AI agent" — 唯一内置学习闭环的 Agent 系统

🏗️ 整体架构

Hermes Agent 代码规模超过 12 万行 Python 代码，核心模块如下：

hermes-agent/
├── run_agent.py          # AIAgent 核心对话循环 (~10800行)
├── cli.py                 # 交互式终端 UI (~10000行)
├── model_tools.py         # 工具发现、schema 收集、分发
├── toolsets.py            # 工具分组和平台预设
├── hermes_state.py       # SQLite 会话存储 + FTS5 搜索
├── agent/                 # Agent 内部模块
│   ├── memory_manager.py     # 记忆管理
│   ├── context_compressor.py # 上下文压缩
│   ├── prompt_builder.py     # 提示词构建
│   └── ...
├── hermes_cli/           # CLI 子命令系统
├── tools/                 # 48 个工具实现
├── gateway/               # 消息平台网关 (15+ 平台)
└── skills/                # Skills 系统

🔥 设计亮点一：自我进化机制

这是 Hermes 与其他 Agent 框架最核心的差异。它不是每次对话都从零开始，而是具备持续学习的能力：

🧠 Agent-Curated Memory

Agent 会周期性「nudge」自己，主动保存重要信息到记忆系统。不是被动记录，而是主动提炼。

⚡ Autonomous Skill Creation

完成复杂任务后，Agent 自动创建对应的 Skill，下次遇到类似任务可以直接调用。

🔄 Skills Self-Improve

Skills 在使用过程中自动优化。每次调用都是学习和改进的机会。

🔍 FTS5 Session Search

跨会话搜索历史对话，配合 LLM 摘要生成，实现真正的事件溯源能力。

核心循环

while api_call_count < max_iterations:
    response = client.chat.completions.create(
        model=model, messages=messages, tools=tool_schemas
    )
    if response.tool_calls:

        messages.append(tool_result_message(result))

    else:

🔌 设计亮点二：Provider 中立架构

Hermes 不绑定任何模型提供商，支持20+ 模型，一键切换：

  • MiniMax / MiniMax China — 国产大模型
  • OpenAI — GPT-4、Claude 等
  • Anthropic — Claude 直接集成
  • OpenRouter — 聚合 200+ 模型
  • Kimi/Moonshot — Moonshot 模型
  • GLM / Z.AI — 智谱/百川
  • 本地模型 — Ollama、vLLM 等

"Switch with hermes model — no code changes, no lock-in."

— Hermes 官方

🛠️ 设计亮点三：Tool Registry 架构

Hermes 的工具系统采用中心化注册表模式，48 个工具通过统一接口注册和调用：

# tools/registry.py — 中心化注册表
registry.register(
    name="example_tool",
    toolset="example",
    schema={"name": "example_tool", "description": "...", "parameters": {...}},
    handler=lambda args, **kw: example_tool(...),
)

# 添加新工具只需 3 步：
# 1. 创建 tools/your_tool.py
# 2. 在 model_tools.py 添加导入
# 3. 在 toolsets.py 添加到工具集

这种设计的优势：高度可扩展，社区可以轻松贡献新工具。

已有工具分类

  • terminal_tool.py — 终端命令执行
  • file_tools.py — 文件读写/搜索/补丁
  • web_tools.py — 网页搜索/提取
  • browser_tool.py — 浏览器自动化
  • mcp_tool.py — MCP 协议客户端
  • delegate_tool.py — 子 Agent 委托
  • code_execution_tool.py — 代码执行沙箱

📚 设计亮点四：Skills 系统

Skills 是 Hermes 的知识模块，采用三级渐进式加载，最大限度节省 token：

Level 0

skills_list() — 返回 Skill 列表，约 3k tokens

Level 1

skill_view(name) — 加载完整 Skill 内容

Level 2

skill_view(name, path) — 加载特定引用文件

SKILL.md 格式

---
name: my-skill
description: Brief description
version: 1.0.0
metadata:
  hermes:
    tags: [python, automation]
    category: devops
---
# Skill Title

## When to Use
## Procedure
## Pitfalls
## Verification

Skills 兼容 agentskills.io 开放标准，社区可以共享和复用。

💬 设计亮点五：多平台消息网关

一个 gateway 进程，支持15+ 平台：

  • 即时通讯：Telegram、Discord、Slack、WhatsApp、Signal
  • 企业协作：钉钉、飞书、企业微信
  • 其他：Matrix、Mattermost、Email、Home Assistant

用户可以在任何平台与 Hermes 对话，跨平台会话历史自动同步。

🖥️ 设计亮点六：多后端执行环境

Hermes 的终端工具支持6 种后端：

  • local — 本地执行
  • docker — 容器隔离
  • ssh — 远程执行
  • daytona — 云端持久化
  • singularity — 高性能容器
  • modal — Serverless（空闲时休眠，几乎零成本）

🔄 设计亮点七：OpenClaw 迁移支持

Hermes 内置OpenClaw 迁移工具，一行命令完成全部迁移：

hermes claw migrate              # 交互式迁移
hermes claw migrate --dry-run    # 预览将要迁移的内容

迁移内容：SOUL.md、Memory、Skills、API Keys、配置、AGENTS.md

💡 对国产 AI Agent 发展的启示

通过研究 Hermes，我认为国产 AI Agent 可以在以下几个方向突破：

1. 自我进化能力

从「每次都是新手」进化到「持续学习」。这需要记忆系统、Skill 自创建、经验沉淀的完整闭环。

2. Provider 中立

不绑定单一模型，支持国产大模型（MiniMax、GLM 等）作为默认选项，降低对海外模型的依赖。

3. 垂直领域 Skills

针对运维、金融、医疗等行业开发专业 Skills，形成垂直壁垒。

4. 企业级网关

深度集成钉钉、飞书、企业微信等国产平台，满足企业合规需求。

🚀 总结

Hermes Agent 的核心设计理念可以概括为：

📌 Hermes 的本质是「Agent 操作系统」

不绑定模型 | 不绑定平台 | 自我进化 | 高度可扩展

对于国内 AI Agent 的发展，Hermes 提供了很好的参考：如何在保证技术先进性的同时，实现真正的工程落地。

下一个问题不是「能不能做」，而是「谁能做得更好用、更稳定、更有价值」。
